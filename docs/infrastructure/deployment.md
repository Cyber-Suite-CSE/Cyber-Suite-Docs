# Platform Deployment Guide

This guide details the Kustomize deployment architecture of the Project Vigilion platform, its configuration variables, and environment overlays.

For installation details of the Kubernetes cluster, ingress configuration, or CI/CD pipelines, see the dedicated sub-guides:
*   [Kubernetes Cluster Setup](./cluster-setup.md) (K3s, Ingress Traefik, and Let's Encrypt certificates)
*   [Continuous Integration & Deployment (CI/CD)](./CI-CD.md) (GitHub Actions pipelines, self-hosted runner service)

---

## 1. Deployment Architecture

The deployment configurations are organized using a standard base-and-overlay architecture:
*   **Base manifests** define the default Kubernetes configurations, network namespaces, persistent volume claims, routing rules, and deployments for all microservices.
*   **Staging/Development Overlay** configures variables for dynamic testing, debugging options, and administrative utilities.
*   **Production Overlay** configures optimized, secure settings tailored for stability and secure access limits.

---

## 2. Environment Configurations

All system variables are injected at deployment time using ConfigMaps and Secrets generated by Kustomize from two environment files.

:::note
**WARNING:** Never check `.secrets.env` or `.config.env` into version control. They contain private credentials, database passwords, and API keys.
:::

### 1. Dynamic Configurations (`.config.env`)
Manages non-sensitive, environment-specific configs, such as Docker image names, service ports, and internal endpoints.

Key variables:
*   **Docker Images**: `API_GATEWAY_IMAGE=csecyber/cyber-suite-api-gateway:latest`, etc.
*   **Service Ports**: `API_GATEWAY_PORT=3000`, `DASHBOARD_PORT=3000`, etc.
*   **Internal URLs**: `WEB_SCANNER_URL=http://web-domain-scanner:8001` (Kubernetes DNS naming).
*   **CORS Configuration**: `ALLOWED_CORS`. Must include the server IP (e.g. `http://<SERVER_IP>`) or domain name (`https://vigilion.cse.mrt.ac.lk`) to prevent browser blocks.

### 2. Sensitive Credentials (`.secrets.env`)
Manages security parameters and private external API keys.

Key variables:
*   **System Security**: `ADMIN_PASSWORD` (dashboard login), `JWT_SECRET` (token signatures), and `MSF_PASSWORD` (RPC password for Metasploit).
*   **External APIs**: `GOOGLE_GENERATIVE_AI_KEY`/`GEMINI_API_KEY` (used for code analysis scanning), `VIRUSTOTAL_API_KEY`, `SHODAN_API_KEY`, and `CENSYS_API_ID`/`CENSYS_API_SECRET`.

## 3. Deploying the Platform

Deploying the Vigilion platform involves the following key steps:
1.  **Transfer Manifests**: The orchestrator configuration folder (`Deployment-Repo`) is uploaded or cloned to the target VM server environment.
2.  **Initialize Environments**: Dynamic and sensitive configurations are prepared in the target environment by duplicating the `.env.example` templates and inserting relevant keys, CORS configuration, and security parameters.
3.  **Execute Overlay Deployment**: Kustomize overlays (e.g., `dev` or `prod`) are applied to the Kubernetes cluster to start the system and its dependent components.
4.  **Verify Pods**: The health check interface or command-line logs are reviewed to confirm that all service pods have initialized and transitioned to a healthy runtime status.

---

## 4. Scaling & Cleanups

*   **Scaling Microservices**: Individual services (such as the SAST scanner engine) can be scaled horizontally to support a higher scan throughput.
*   **Teardown**: Deployed configurations can be removed, stashed, or destroyed to clean up resources and free memory allocation on the server when needed.
